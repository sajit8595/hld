Section 02

The 5 Estimation Formulas

Every back-of-envelope calc reduces to one or more of these. Apply in order: Throughput → Storage/Day → Storage/5yr → Cache → Bandwidth.

Formula A — Throughput (RPS)
// Formula
RPS = Daily Requests ÷ 86,400 sec ≈ Daily Requests ÷ 10⁵
// Hint: 1 day = 100,000 seconds (round 86,400 → 10⁵)
// Separate read RPS and write RPS using the read:write ratio
// Example: 100M DAU, each generates 1 action/day
Write RPS = 100M ÷ 10⁵ = 1,000 RPS (1K RPS)
Read RPS = 1K × read:write ratio (e.g. 100:1) = 100K RPS
⚡ Peak multiplier: Always apply 3× to average RPS for peak traffic. System must handle peaks, not averages. → Peak write RPS = 3K, peak read = 300K.
Formula B — Storage Per Day
// Formula
Storage/Day = Daily Writes × Object Size (in bytes)
// Convert final answer: 10⁹ bytes = 1 GB, 10¹² bytes = 1 TB
// Example: 100M writes/day, object size = 100 B (short text)
Storage/Day = 10⁸ × 10² = 10¹⁰ bytes = 10 GB/day
// If mixed content (80% text, 20% images):
= (80M × 100B) + (20M × 100KB) = 8GB + 2,000GB ≈ ~2 TB/day
Formula C — Storage / 5 Years (Total)
// Formula
Storage/5yr = Storage/Day × 2,000 days (5yr ≈ 2×10³ days)
// Add replication factor (usually 3×) for final disk requirement
// Example (from Formula B, 10 GB/day)
Raw storage = 10 GB × 2,000 = 20 TB
With 3× replication = 20 TB × 3 = 60 TB disk provisioned
💡 Tiered storage: Hot data (last 30 days) → SSD. Warm (30d–1yr) → HDD. Cold (>1yr) → Glacier/S3 IA. Mention this — it shows cost awareness.
Formula D — Cache (RAM) Required
// Formula (80/20 rule: 20% of content drives 80% of reads)
Cache RAM = Daily Reads × Object Size × 0.20
// This is the minimum hot-set cache size to achieve ~80% hit rate
// Example: 10B daily reads, object size = 100B
Cache = 10¹⁰ × 10² × 0.2 = 2 × 10¹¹ bytes = 200 GB RAM
// This exceeds single Redis node → use Sharded Redis Cluster
📌 Threshold: < 100 GB → single Redis node (modern servers have 128–256 GB RAM). ≥ 100 GB → sharded Redis cluster. State this explicitly.
Formula E — Network Bandwidth / Egress
// Formula
Bandwidth = Peak Concurrent Users × Avg Bitrate per User
// Convert bits → bytes: ÷ 8. Convert to GB/s: ÷ 10⁹
// Example: Netflix — 10M concurrent users, 4 Mbps per stream
Egress = 10M × 4 Mbps = 40 Tbps ÷ 8 = 5 TB/sec data egress
// 5 TB/sec >> 1 Gbps threshold → CDN is mandatory (not optional)
🌐 CDN threshold: If egress > 1 Gbps from your origin, you must use a CDN. Netflix Open Connect, Cloudflare, CloudFront — mention the type based on content.

Section 03

Requirements Gathering

Hello Interview framework: drive requirements before touching the whiteboard. Interviewers give seniority credit for good clarifying questions — this is where FAANG gaps are decided.

The 3-Phase Requirements Framework
1
Functional Requirements — What must the system do?
Define the core user actions (MVP scope). Explicitly list what's out of scope. Aim for 2–4 core features, not a full product. Misscoping is the #1 reason candidates fail design rounds.
2
Non-Functional Requirements — What qualities must the system have?
Scale, latency, availability, consistency, durability. These directly determine your architecture choices. Without NFRs, every architecture decision is unjustified.
3
System-Specific Constraints — What is unique about this domain?
Compliance (GDPR, PCI-DSS), ordering guarantees, media types, real-time requirements, geographic constraints. These create the "deep dive" topics.
Functional Requirements — Questions to Ask
📌 "What are the core features for MVP scope? What can we explicitly leave out?"
Sets scope. Prevents you from designing every feature of a full product.
📌 "Who are the primary users? Is there a producer/consumer split (e.g., content creators vs readers)?"
Determines read vs write ratios and whether fan-out is needed.
📌 "What does a successful core interaction look like end-to-end? Walk me through it."
Clarifies the critical path you need to design for.
📌 "Are there user roles (admin, moderator, viewer)? Do they affect data access patterns?"
May introduce ACL/RBAC complexity into the design.
Non-Functional Requirements — Critical Questions
⚙️ Scale: "How many DAU? What is the expected peak concurrent users? What's the read:write ratio?"
Drives: DB choice, sharding, caching strategy. Different by 10× = different architecture.
⚙️ Latency: "What is the acceptable P99 latency for the critical read path? And write path?"
Drives: In-memory vs SSD, caching layer, synchronous vs async writes.
⚙️ Availability: "What is the acceptable downtime? Is this 99.9% or 99.99%? Any SLA contractual obligations?"
Drives: Multi-AZ, active-active vs active-passive, circuit breakers, replication factor.
⚙️ Consistency: "Can users see slightly stale data? Or is strong consistency required (e.g., for payments)?"
Drives: SQL vs NoSQL choice. Eventual OK → Cassandra/DynamoDB. Strong required → SQL.
⚙️ Durability: "Is any data loss acceptable? (e.g., last 5 seconds of metrics vs. financial transactions)"
Drives: Write path (sync vs async replication), WAL, fsync settings, backup strategy.
⚙️ Geo: "Is this a single region? Multi-region active-active? Or global CDN with regional writes?"
Drives: Global load balancing, data sovereignty, replication topology, latency targets.

Add cache layer ≥ 100K read RPS RPS is the only driver for caching Even at 1K RPS, if DB P99 > your latency budget (e.g. 50ms), or if 80% of reads hit the same 5% of data (hot working set), cache is mandatory regardless of RPS "Add cache when: (a) hot working set exceeds DB buffer pool, OR (b) DB P99 exceeds latency SLO, OR (c) read RPS > 100K. Shard the database ≥ 1 TB stored Storage size drives sharding decision Hot working set > RAM buffer pool (typically 25% of server RAM) → disk I/O on reads. A 500GB table with a 100GB hot set on a 128GB RAM server forces disk reads constantly, even though total storage < 1TB "Shard when: hot working set exceeds buffer pool (≈25% of DB server RAM). Rule: 1TB on 512GB RAM server is fine; 1TB on 64GB RAM server needs sharding or read replicas."
Use CDN ≥ 1 Gbps egress Bandwidth is the only CDN trigger Even at 100 Mbps, if users are globally distributed (US + EU + APAC), CDN reduces P99 by 60–80% by serving from edge PoPs. Geographic latency is an independent trigger. "CDN when: (a) egress > 1 Gbps (cost/capacity), OR (b) users span > 1 geographic region (latency). Both are independent conditions."

The real drivers for cluster: (a) multi-key ops across slots won't work, (b) need zero-downtime re-sharding, (c) need to isolate hot keys to separate nodes "Use Redis Cluster when you need: horizontal scaling beyond 250 GB, hot-key isolation, or live resharding. Use Sentinel (single node + failover) up to 250 GB — simpler ops."
Use NoSQL over SQL ≥ 10K write RPS Write RPS is the primary signal DynamoDB at 1K RPS is justified if schema is truly key-value (no joins ever needed). SQL at 50K RPS is fine if workload is read-heavy with good indexing. Consistency requirement > throughput as a decision driver. "Choose NoSQL when: access pattern is key-value (no joins), schema is known at design time, AND writes > 10K OR you need auto-scaling. Don't pick NoSQL just for write volume if consistency or query flexibility is needed."

PACELC — More Accurate Than CAP for Modern Systems
CAP Theorem (What You Know)
During a Partition: choose Consistency or Availability. Only 2 of 3.

Problem: network partitions are rare. Systems spend 99.99% of their time in normal operation with no partition. CAP says nothing about that case.
PACELC (The Real Trade-off)
Partition: choose Availability or Consistency (same as CAP)
Else (normal operation): choose Latency or Consistency

Even without partition: synchronous replication = consistent but high latency. Async replication = low latency but stale reads. This is the trade-off that actually matters day-to-day.
SystemPartition choiceNormal ops choiceReal implication
DynamoDBAPEL (low latency)Eventually consistent reads by default. Strongly consistent reads cost 2× RCUs and add latency.
CassandraAPEL (tunable)QUORUM reads add 1 extra node hop vs ONE. Tune per operation — not per cluster.
Postgres (sync replica)CPEC (high consistency)Synchronous commit waits for standby ACK → adds ~1ms per write. Highest consistency, highest write latency.
Postgres (async replica)CPELLeader commits immediately, replica may lag 50ms–1s. Reads from replica can be stale. Most common production setup.
Spanner / CockroachDBCPECTrueTime / HLC clocks for external consistency. Adds ~7ms commit latency. Only justified when global strong consistency is mandatory.
💡 Interview line: "CAP applies during partitions which are rare. The everyday trade-off is PACELC — latency vs consistency in normal operation. Cassandra with QUORUM gives strong consistency but adds one round-trip. Async Postgres replica is low latency but may serve 50ms stale reads. I'd choose based on the SLO, not just the CAP label."

Fan-out — News Feed / Timeline Delivery

Scalability Pattern · Used in: Twitter, Instagram, Facebook Feed, TikTok
+

Problem: User A posts content. A's 1M followers should see it in their feed. How do you deliver one write to N readers efficiently without melting your DB?

Two Strategies
Fan-out on Write (Push)
When user posts, immediately write post_id to every follower's inbox (Redis sorted set, score = timestamp).

Pros: O(1) feed reads — just ZRANGE inbox.
Cons: Write amplification. Justin Bieber (100M followers) = 100M Redis writes per post.
Fan-out on Read (Pull)
When user opens app, fetch post IDs from all followed users and merge/rank them.

Pros: No write amplification, always fresh.
Cons: O(following_count) DB reads per feed load. 1,000 follows = 1,000 queries.

No write amplification for celebrity accounts
Personalization: inline ads/recommendations easily added

Redis inbox storage: 500M users × 20 posts × 8 bytes ≈ 80 GB RAM
💡 Interview Pro Tip
"I'd define a celebrity threshold of ~10K followers. Below that, fan-out on write; above that, skip fan-out and compute on read using a dedicated celebrity cache. This solves the write amplification bottleneck that breaks pure push models at scale."

Distributed Caching — Strategies & Invalidation

Performance Pattern · Used in: Every high-scale system
+

Problem: DB reads are the bottleneck. 1ms SQL vs 0.1ms Redis. At 100K RPS, every DB round-trip is a potential thundering herd. Three patterns for how cache and DB stay in sync.

flowchart LR
    subgraph CA[1 Cache-Aside lazy]
        direction TB
        RA([Read]) --> HC{hit?}
        HC -- yes --> RET1[Return]
        HC -- no --> DBQ[(DB)]
        DBQ --> CSET[Cache SET TTL]
        CSET --> RET1
        WA([Write]) --> DBW[(DB)]
        DBW --> CDEL[Cache DEL\ninvalidate]
    end
    subgraph WT[2 Write-Through sync]
        direction TB
        WB([Write]) --> BOTH[DB + Cache\nsync write]
        RB([Read]) --> CHIT[Always cache hit]
    end
    subgraph WBK[3 Write-Behind async]
        direction TB
        WC([Write]) --> CONLY[Cache only]
        CONLY --> FAST[Return fast]
        CONLY -.->|batch flush 1-5s| DBC[(DB)]
    end
1
Cache-Aside (Lazy Loading) — Most Common
App checks cache → miss → fetch from DB → write to cache → return. Cache only populated on demand. TTL controls staleness.
Read: cache.get(key) → miss → db.query() → cache.set(key, value, TTL) → return Write: db.write(data) → cache.delete(key) ← invalidate, don't update Risk: Cache miss storm → many concurrent requests all miss → DB overwhelmed Fix: Mutex lock on first miss (only 1 request fetches from DB; others wait)
2
Write-Through — Consistency First
Write to cache AND DB synchronously in same transaction. Cache always has latest value. Higher write latency (2 writes), but never stale reads.
Write: db.write(data) AND cache.set(key, value) ← synchronous Read: cache.get(key) → always a hit for recently written data Risk: Cache churn if data is written but never read (wasted write) Use: Configuration, user profile, read-heavy after every write
3
Write-Behind (Write-Back) — Throughput First
Write to cache only, then async batch-flush to DB. Very fast writes; risk of data loss if cache crashes before flush.
Write: cache.set(key, value) → return immediately ← fast! async: batch flush dirty keys → DB (every 1-5 seconds) Risk: If cache crashes before flush → data lost permanently Use: High-write systems where some loss is acceptable (counters, views)
When to Cache
Data is read much more than written (100:1 ratio)
Computation is expensive (aggregations, rankings)
Slight staleness is acceptable (social counters)
Cache Pitfalls
Thundering herd: cache key expires, 10K requests hit DB simultaneously
Cache penetration: request for non-existent key (store null with TTL)
Hot key: 1 key gets millions of requests (shard key or local cache)
💡 Interview Pro Tip
"I'll use cache-aside with TTL for this read-heavy path. For the thundering herd problem, I'd use a distributed mutex — the first request acquires a lock and fetches from DB while others wait. For hot keys like trending posts, I'd add local in-process LRU cache on app servers to further reduce Redis load."

Message Queue & Async Processing

Decoupling Pattern · Used in: Every large-scale system (Kafka, SQS, RabbitMQ)
+

Problem: Service A needs to trigger work in Services B, C, D. If A calls them synchronously, A's latency = sum of all. If B is down, A fails. Message queues decouple producers from consumers.

1
Identify async candidates
Any task that doesn't need a synchronous response: image resizing, email sending, notification dispatch, search indexing, analytics logging, thumbnail generation.
2
Design the event schema
Events must be self-contained (include all needed data). Always include: event_id (for deduplication), timestamp, version, and payload. {event_id, type, user_id, data, created_at}
3
Consumer group design
Multiple services can independently consume same topic. Kafka: each service gets its own consumer group, reads from offset independently. Enables replay for new consumers.
4
Handle failures (DLQ)
Failed messages retry with exponential backoff (1s, 2s, 4s, 8s…). After N retries, route to Dead Letter Queue for manual inspection. Never silently drop events.

Kafka (Choose When)
Need event replay / audit log
Multiple independent consumers of same event
Very high throughput (> 1M events/sec)
Need to retain events for days/weeks

Simple task queue (one consumer per message)
Need managed service with minimal ops
FIFO ordering required per-entity (SQS FIFO)
RabbitMQ: complex routing rules needed
💡 Interview Pro Tip
"Consumers must be idempotent — include an event_id and check for duplicates before processing. Kafka provides at-least-once delivery; exact-once processing requires deduplication logic on the consumer side. I'd use a Redis SET with the event_id as key, TTL 24h."

Rate Limiting — Token Bucket, Sliding Window, Leaky Bucket

Protection Pattern · Used in: API Gateways, Auth Services, Notification Systems
+

Problem: Prevent abuse, protect downstream services, enforce fair use per user/IP/API key. Three algorithms with different burst vs smoothness characteristics.

flowchart LR
    subgraph TB[Token Bucket burst OK]
        direction TB
        R1([Request]) --> LUA1{tokens ge 1?}
        LUA1 -- yes --> DEC[DECR tokens\nLua atomic]
        DEC --> A1([200 Allow])
        LUA1 -- no --> REJ1([429 Reject])
        REF([Scheduler]) -->|add R tokens/sec cap max| LUA1
    end
    subgraph SW[Sliding Window accurate]
        direction TB
        R2([Request]) --> ZADD[ZADD ts:now]
        ZADD --> ZREM[ZREM old entries]
        ZREM --> CNT{count gt limit?}
        CNT -- yes --> REJ2([429])
        CNT -- no --> A2([200 Allow])
    end
    subgraph LB[Leaky Bucket smooth]
        direction TB
        R3([Request]) --> QUE[Queue]
        QUE -->|drain at fixed rate| A3([Process])
    end
1
Token Bucket (Most Common in APIs)
Bucket holds max N tokens, refills at R tokens/sec. Each request consumes 1 token. Allows bursting up to bucket size, then enforces average rate.
Redis implementation: tokens = redis.GET("rl:{user_id}") if tokens >= 1: redis.DECRBY("rl:{user_id}", 1) redis.EXPIRE("rl:{user_id}", window) allow request else: reject with 429 Too Many Requests Refill: Lua script (atomic): calculate elapsed time → add tokens → cap at max
2
Sliding Window Counter (Most Accurate)
Track request timestamps in a sorted set. Count entries in the last N seconds. Expensive but no boundary burst problem of fixed windows.
Redis implementation: ZADD rl:{user_id} {now_ms} {request_id} # add entry ZREMRANGEBYSCORE rl:{user_id} 0 {now - 60s} # expire old count = ZCARD rl:{user_id} if count > limit: reject else allow EXPIRE rl:{user_id} 60
3
Leaky Bucket (Smoothest Output)
Queue incoming requests; process at fixed rate. Smooths bursts. Ideal for rate-limiting outgoing calls (e.g., sending notifications at max N/sec to FCM).
Distributed Rate Limiting
Centralized Redis: all API nodes hit same Redis → consistent, ~0.5ms overhead per request. Works well up to ~500K RPS.

Distributed approx: each node has N/server_count quota, sync every second. Trade-off: over-counting possible but extremely fast.
Rate Limit Keys
Choose granularity based on use case:

user:{id} → per-user limits
ip:{addr} → anonymous protection
api_key:{key} → tenant limits
endpoint:{path}:{user} → per-route limits
global → service-level protection
💡 Interview Pro Tip
"I'd use Token Bucket via a Redis Lua script for atomicity (GET + DECR in one network round-trip). The rule: 1,000 req/min per user, burst up to 100 req/sec. For distributed systems, centralized Redis adds ~0.5ms per request — acceptable. Return headers: X-RateLimit-Remaining, X-RateLimit-Reset."

Consistent Hashing & Sharding Strategies

Scalability Pattern · Used in: Redis Cluster, Cassandra, DynamoDB, CDNs
+

Problem: With N servers, hash(key) % N means adding/removing a server remaps ~all keys → cache invalidation, DB resharding. Consistent hashing remaps only ~1/N keys.

1
Hash Ring Construction
Map servers and keys to same 0→2³² circular ring using consistent hash function. Key maps to the first server clockwise. Adding a server only captures keys from its successor.
flowchart LR
    KEY([key]) --> HASH[SHA256\nmod 2^32]
    HASH --> RING{Walk ring\nclockwise}
    RING --> S1[Server S1\nvnodes 0-85]
    RING --> S2[Server S2\nvnodes 86-170]
    RING --> S3[Server S3\nvnodes 171-255]
    ADD([Add Server S4]) --> VN[256 vnodes\nassigned to S4]
    VN --> REM[Only 1/N keys\nremapped from S2]
    REM --> CHEAP([No full\nresharding!])
    style CHEAP fill:#064e3b,stroke:#10b981,color:#e2e8f0
    style REM fill:#064e3b,stroke:#10b981,color:#e2e8f0
Ring: 0 ─────────── S1 ──────────── S2 ──────────── S3 ─── 2³² Keys: k1(→S1) k2(→S2) k3(→S2) k4(→S3) Add S4 between S1 and S2: Only keys between S1 and S4 move from S2 → S4 (~1/N remapped) Everything else stays: k3 still → S2, k4 still → S3
2
Virtual Nodes (Vnodes) — Solve Uneven Distribution
Each physical server gets K virtual nodes on the ring (e.g., K=150). More vnodes = more uniform distribution. Cassandra uses 256 vnodes per node by default.
3
Choose Shard Key Carefully
Good: user_id (uniform distribution, single-user queries). Bad: country_code (hot spots), created_at (all writes on latest shard). Hot key detection → split hot shards or add read replicas.
Sharding Strategies
Hash sharding: uniform, but range queries span all shards
Range sharding: good for range queries, uneven load risk
Directory sharding: flexible routing, lookup table required
Geo sharding: data residency, low cross-region latency
Sharding Pitfalls
Cross-shard queries: JOIN across shards = N DB calls + app-level merge
Cross-shard transactions: use Saga or 2PC (avoid if possible)
  • Rebalancing: online resharding (DynamoDB does this; SQL databases do not)
    • 💡 Interview Pro Tip
    "I'd shard by user_id hash to keep all of a user's data on one shard — simplifies single-user queries and avoids cross-shard lookups for the critical path. I'd use 256 virtual nodes per server for uniform distribution, the same approach as Cassandra."

    Search & Typeahead Autocomplete

    Data Pattern · Used in: Google Search, Twitter Search, Uber Destination, Amazon Product Search
    +

    Problem: Users expect sub-100ms search results across billions of documents. Two sub-problems: full-text search (Elasticsearch) and typeahead suggestions (trie / Redis sorted sets).

    Full-Text Search (Elasticsearch)
    1
    Inverted Index
    For each word, maintain a posting list: word → [doc_id, position, frequency]. Query tokenizes input → intersects posting lists → ranks by BM25 relevance score.
    2
    Write Path — Mirror from Primary DB
    Never write directly to ES as source of truth. Use CDC (Change Data Capture via Debezium/Kafka) to stream changes from primary DB → ES. Or direct write + async index.

    HYD --> RESP([Response])

    Typeahead / Autocomplete
    1
    Precomputed Top-K per Prefix (Practical Production Approach)
    Maintain a map: prefix → [top-5 suggestions by search frequency]. Updated by batch job every 5–10 minutes from query logs. Stored in Redis: HGET suggestions:{prefix}.

    Schema mapping changes require full re-index
    Deep pagination (page 1000) is slow — use search_after

    Geospatial Indexing — Nearby Search

    Data Pattern · Used in: Uber, Lyft, DoorDash, Google Maps, Yelp
    +

    Problem: Find N nearest drivers/restaurants/POIs to a given lat/lng in < 20ms. Can't do sqrt((x2-x1)² + (y2-y1)²) across 500K live drivers every request.

    Geohash
    Divide world into grid cells, encode as base32 string. Longer hash = smaller cell. Nearby cells share prefix. Redis GEOADD/GEORADIUS uses geohash internally.

    Precision: 6 chars = ~1.2km², 7 chars = ~152m², 8 chars = ~38m²

    Edge case: Two points in adjacent cells may not share prefix — always search neighbor cells too.
    Quadtree
    Recursively split 2D space into 4 quadrants. Split when > K entities in a cell. Efficient for dynamic data (drivers constantly moving).

    Best for: In-memory structure for real-time driver lookup.

    Operations: Insert O(log n), Range query O(log n + k), Updates cheap for moving entities.

    Notification System — Multi-Channel Reliable Delivery

    Integration Pattern · Used in: Every consumer app (Facebook, WhatsApp, Uber)
    +

    Problem: Send notifications across push (iOS/Android), email, and SMS. Must handle delivery failures, user preferences, rate limits, and at-least-once guarantees without duplicates.

    1
    Event Ingestion & Priority Queuing
    All system events enter via Notification Service. Classify by priority: P0 (account alerts, OTP), P1 (social interactions), P2 (marketing). Use separate Kafka topics or priority queue lanes.
    2
    User Preference Check
    Before sending, check user_preferences DB: which channels enabled, DND hours, notification frequency cap. Skip silently if user opted out. Cache preferences in Redis (30 min TTL).
    3
    Template Rendering & Channel Dispatch
    Render message from template DB. Route to channel workers: FCM/APNs for push, SES/SendGrid for email, Twilio for SMS. Each worker retries independently with exponential backoff.
    4
    Deduplication & DLQ
    Idempotency key: hash(user_id + notification_type + time_window). Check Redis SET before sending. After max retries → DLQ → alert on-call. Track delivery status in DB.
    flowchart LR
    EVT([Event\nlike/order/alert]) --> NS[Notification\nService]
    NS --> DUP{Redis dedup\ncheck key}
    DUP -- duplicate --> DROP([Drop silently])
    DUP -- new --> PQ[(Kafka\nP0 P1 P2\npriority queues)]
    PQ --> PREF{User prefs\nDND check?}
    PREF -- opted-out --> SKIP([Skip])
    PREF -- ok --> TMPL[Template\nRenderer]
    TMPL --> PUSH[FCM/APNs\nworker]
    TMPL --> EMAIL[SES/SendGrid\nworker]
    TMPL --> SMS[Twilio\nworker]
    PUSH --> DEV([Device])

    Matching with Timeout Cascade — No Rider Left Waiting
    Problem: Driver A doesn't respond in 10s. Rider must be matched without perceiving the failure.

    Trip State Machine — Idempotent Transitions Prevent Double Billing
    Problem: Network glitch sends "trip complete" event twice → PaymentService charges rider twice.

PaymentService uses idempotency key hash(trip_id + "settle") checked in Redis SET (TTL 24h) before any charge attempt.
    !
    Geospatial Sharding for 10M+ Global Drivers
    Problem: Single Redis GEO node tops out around 500K–1M entries with good GEORADIUS latency. 10M global drivers across 500 cities needs sharding.

City Registry is tiny: 10K cities × 50B = 500 KB — lives in every service's local memory, refreshed every 60s.

    Redis GEO: 50MB for 500K drivers, sub-ms GEORADIUS queries
    MySQL for trips: 200 RPS is trivial; ACID guarantees billing safety

    Stateless Matching Service scales horizontally without coordination

    Redis GEO data is ephemeral: driver crash → position lost until next ping (4s gap)
    OSRM routing calls add 20–50ms per match — cache routing results by geohash cell

    !
    ID Generation: Token-Range Approach (Best Answer)
    Option A — MD5(long_url) (wrong): Same URL always maps to same code (no per-user versions), hash collisions in first 7 chars are real, can't guarantee uniqueness across distributed nodes.

    Option B — Auto-increment DB counter (simple but brittle): Sequential codes are guessable (security). Single DB is SPOF. Works only up to ~5K RPS write.

    Option C — Token range (✓ production answer): Each URL Service instance calls Counter DB once: "give me IDs 5,000,001–6,000,000." Stores range locally, increments atomically without coordination for the next 1M requests. Counter DB is only called every 1M IDs — near-zero overhead. Thread-safe via AtomicLong. Convert ID to base62 = 7-char code. This is how Instagram Snowflake and Flickr ticket servers work.
    !
    Bloom Filter — Eliminate 99.9% of Collision Checks
    Problem: On every write, must verify the generated code doesn't already exist in the DB. 1K RPS × 1 DB read = 1K extra reads/sec just for collision checking — wasteful when collisions are astronomically rare.

    Solution: In-memory Bloom filter loaded at startup (all existing codes). False-positive rate ≈ 0.1% with ~1 MB per 10M entries. Before DB write: if Bloom says "definitely not in DB" → skip DB check, just insert. Only the 0.1% false-positives trigger a real DB lookup. At 10B total URLs: 1GB Bloom filter, 99.9% of writes skip the existence check entirely. When a new URL is inserted → add to Bloom filter.
    !
    301 vs 302 — The Hidden Tradeoff Every Interviewer Wants
    301 Permanent Redirect: Browser caches it permanently. After first visit, browser goes directly to destination — your servers see zero traffic for that link ever again. Zero analytics. Cannot change destination. CDN caches forever (until explicit purge).

    302 Temporary Redirect: Browser never caches. Every click hits your redirect servers. Enables: click counting, A/B destination testing, expiry enforcement, geography-based routing. But 100× the server load vs 301.

    Production answer: Use 302 by default (analytics, expiry enforcement). Offer 301 as a "permanent link" premium feature. Mention this explicitly — it shows you understand the caching layer implications.
    Strengths
    Token-range ID gen: zero per-request coordination, no SPOF, trivially scalable
    Bloom filter: eliminates 99.9% of DB existence reads on write path
    CDN edge for hot 301 links: sub-5ms globally, zero origin load
    LRU Redis cache: 2GB covers 50%+ of redirect traffic
    Tradeoffs to State
    301 vs 302: analytics accuracy directly trades off with server load
    Custom aliases: synchronous DB uniqueness check required (no Bloom shortcut)
    Link deletion: CDN-cached 301s are nearly impossible to invalidate globally
    Bloom filter: false positives (0.1%) cause unnecessary DB reads; Bloom never shrinks without rebuild

    Resumable Uploads — TUS Protocol for 10 GB Files
    Problem: A 10 GB upload on mobile fails at 9.9 GB. User must restart from zero — terrible UX.

    Solution (TUS-like protocol):
    1. Initiate: POST /uploads/initiate → server creates upload_id, stores {total_size, chunks_expected} in Redis. Returns upload_id.

    3. Resume after failure: GET /uploads/{id}/status → returns {received_chunks: [0,1,2,4,5]}. Client detects missing chunk 3 → resumes from chunk 3 only.
    4. Complete: After all chunks received → POST /uploads/{id}/complete → S3 CompleteMultipart → trigger transcoding.
    Key: S3 Multipart Upload supports exactly this — parts can be uploaded independently and assembled server-side.

    Node assignment (256 virtual nodes per physical node): key → SHA-256 → 32-bit hash → position on ring → next N nodes clockwise own this key Replication (N=3 default): Coordinator node → replicate put() to next 2 nodes clockwise (RF=3 total) Write acknowledged after W replicas confirm (W=2 default) Read confirmed after R replicas respond (R=2 default) Invariant: W + R > N → QUORUM → at least 1 replica always has latest value Tunable consistency examples: W=1, R=1 → eventual consistency, max throughput (DynamoDB default) W=2, R=2 → strong consistency for N=3 cluster (recommended) W=3, R=1 → sync replication on write, fastest reads W=1, R=3 → async writes, always-consistent reads Conflict resolution on network partition (split brain): Option A: Last-Write-Wins (LWW) using wall clock timestamp → simple, may lose writes Option B: Vector clocks → each write tagged {node_id → version}; concurrent writes surfaced → application resolves conflict (e.g., Amazon shopping cart: union of items) Option C: CRDTs → data types that merge automatically (counters, sets, registers)
    LSM Tree Strengths
    Sequential writes to WAL: NVMe delivers 3 GB/sec sequential vs 100 MB/sec random
    100K+ writes/sec per node: MemTable absorbs bursts, WAL ensures durability
    Bloom filter: non-existent key lookup costs 0 disk I/O (sub-ms miss detection)
    Immutable SSTables: no in-place updates, crash-safe, easy to replicate as files
    LSM Tree Tradeoffs
    Read amplification: cold key may check multiple SSTables (mitigated by Bloom + indexes)
    Write amplification: data written multiple times during compaction (3–10× typical)
    Space amplification: stale versions accumulate until compaction — 2× disk vs actual data size
    Compaction I/O spikes: background merges can saturate disk during heavy compaction rounds

    CQRS + Event Sourcing — Separate Read and Write Models

    Architecture Pattern · E-commerce, Banking, Event-driven Microservices
    +

    Problem: Your write model needs normalized SQL (integrity, ACID). Your read model needs denormalized pre-joined views (sub-10ms reads). They're fundamentally incompatible on one DB.

    1
    When to Use CQRS (and When NOT To)
    USE WHEN: read:write ratio > 10:1, read queries require complex joins across write tables, different teams own read vs write paths. DO NOT USE for simple CRUD apps — it adds 2× codebases, async consistency, and debugging complexity. Over-engineered CQRS is an anti-pattern.
    2
    Command Side — Normalized, Consistent
    Commands mutate state. Each command generates a domain event: OrderPlaced, PaymentProcessed, ItemShipped. Events stored in Event Store (Kafka or EventStoreDB) as the source of truth. The write DB stays lean — just enough to validate commands against current state.

    Event consumers build read projections optimized for specific queries: OrderDashboard (joins order + user + items), UserActivity (recent actions feed), InventoryView (available stock). Stored in whatever fits: Redis, Elasticsearch, MongoDB, or read-optimized SQL tables.

    Saga Pattern — Distributed Transactions Without 2PC

    Resilience Pattern · E-commerce Orders, Travel Booking, Microservice Coordination
    +

    Problem: Order flow: reserve inventory → charge payment → create shipment. If payment fails after inventory reserved, how do you roll back across 3 independent microservices? 2PC (Two-Phase Commit) requires global lock, blocks on partition, and fails catastrophically when coordinator crashes.

    Choreography Saga (Event-driven)
    Each service reacts to events and emits success/failure events. No central coordinator. Compensating transactions roll back previous steps.

    Pros: Fully decoupled, no single coordinator SPOF.
    Cons: Hard to trace which saga a service is in; no single place to see overall progress.
    Orchestration Saga (Central coordinator)
    SagaOrchestrator explicitly tells each service what to do and tracks state machine.

    Pros: Easy to trace, single place to see saga progress, compensations clearly defined.
    Cons: Orchestrator becomes a bottleneck; needs its own HA design.

    Saga: eventual consistency, no locks held across services, works across independently deployed services
    Saga: each step committed independently — intermediate states are visible (dirty read risk)

    2PC: coordinator crash during "prepare" phase → all participants blocked indefinitely (blocking protocol)
    Saga Pitfalls
    Compensations can fail too → need retry (exponential backoff) + human DLQ for stuck sagas
    Isolation: another order can see partially-reserved inventory during saga execution
    Idempotency: every step and compensation must be idempotent — double-execution = same result
    Orchestrator SPOF: needs its own HA design (active-passive with DB-backed state)

Every step must be idempotent — I'd store saga state in a durable DB and use Kafka for event delivery with at-least-once + deduplication."

    Circuit Breaker + Bulkhead — Preventing Cascading Failures

    Resilience Pattern · Netflix Hystrix, Resilience4j, every production microservice mesh
    +

    Problem: Service A calls Service B. B starts timing out (overloaded). A's threads pile up waiting for B responses → A exhausts its thread pool → A becomes unresponsive → upstream services calling A start failing too. One slow service melts the entire system — cascading failure.

    1
    Circuit Breaker — Three-State Automaton
    CLOSED (normal): All requests pass through. Track failure rate in sliding window (last 60s). If failure rate exceeds threshold (e.g., 50%) → trip to OPEN.
    OPEN (fail fast): All requests fail immediately with fallback — no call to Service B. After 30s timeout → transition to HALF-OPEN.
    HALF-OPEN (probe): Allow 1 request through. Success → CLOSED (recovered). Failure → OPEN again (still down, extend isolation).
    stateDiagram-v2
    [*] --> CLOSED
    CLOSED --> CLOSED : failure_rate lt 50pct
    CLOSED --> OPEN : failure_rate ge 50pct in 60s
    OPEN --> HALFOPEN : after 30s timeout
    OPEN --> OPEN : all requests fallback cache
    HALFOPEN --> CLOSED : probe success recovered
    HALFOPEN --> OPEN : probe failure still down
    Circuit Breaker state machine: CLOSED → failure_rate < 50% in 60s window → stay CLOSED CLOSED → failure_rate ≥ 50% → OPEN (trip) OPEN → all requests: return fallback immediately (no network call) → after 30s → HALF-OPEN HALF-OPEN → send 1 probe request to Service B → success → CLOSED → failure → OPEN (reset 30s timer) Fallback strategies (in priority order): 1. Return cached response from Redis (stale but functional) 2. Return empty/default response (degraded functionality) 3. Queue request for later processing (async retry) 4. Return user-friendly error ("try again in a moment")
    2
    Bulkhead — Thread Pool Isolation per Dependency
    Limit concurrent calls to each downstream service independently. If Service B is slow, only B's thread pool fills — calls to Services C and D are unaffected. Named after ship compartments that prevent one flood from sinking the entire vessel.
    Without Bulkhead: Shared thread pool: 100 threads total Service B slow → 100 threads blocked on B → Service C, D starved → system-wide failure With Bulkhead (thread pool per dependency): Service B pool: 20 threads max → timeout → fail fast (B isolated) Service C pool: 30 threads max → unaffected by B's failure Service D pool: 20 threads max → unaffected Combined (Hystrix/Resilience4j): Bulkhead isolates thread pool PER dependency Circuit Breaker trips per dependency based on failure rate → Two layers: isolation (bulkhead) + fast-fail (circuit breaker)
    Benefits
    Prevents cascading failure: one slow service cannot take down entire system
    Fast failure beats slow failure: user gets instant error vs 30s timeout wait
    Gives downstream services recovery time without traffic pressure
    Fallback responses maintain partial functionality during outages
    Tradeoffs
    Threshold tuning: too sensitive → trips on transient errors (noisy); too loose → too slow to protect
    HALF-OPEN risk: single probe failure re-trips circuit — consider allowing N probes before OPEN
    Stale fallback: if outage lasts hours, cached fallback data becomes dangerously stale
    Overhead: per-request failure counting adds ~0.5ms latency — acceptable for most use cases
    💡 Interview Pro Tip
    "I'd add a circuit breaker on every synchronous external call. Configuration: trip if >50% of requests fail in a 60-second sliding window; OPEN for 30 seconds; fallback serves Redis-cached response. Bulkhead: 20-thread pool per downstream service. This two-layer protection (isolation + fast-fail) is exactly what Netflix invented Hystrix for — and it's now standard in Resilience4j for Java and circuit-breaker libraries in every major language."

    All nodes compete simultaneously: MySQL (atomic compare-and-swap via conditional UPDATE): INSERT INTO leader_lease (service, node_id, expires_at) VALUES ('scheduler', 'node-3', NOW() + INTERVAL 30 SECOND) ON CONFLICT (service) DO UPDATE SET node_id = EXCLUDED.node_id, expires_at = EXCLUDED.expires_at WHERE leader_lease.expires_at < NOW() ← only take over if lease expired Result: exactly one node wins the INSERT/UPDATE (DB serializes concurrent attempts) Leader heartbeat (runs every 10s): UPDATE leader_lease SET expires_at = NOW() + 30s WHERE service='scheduler' AND node_id='node-3' rows_affected = 0 → I've lost leadership! Stop acting as leader immediately. Follower check (runs every 5s): SELECT node_id, expires_at FROM leader_lease WHERE service='scheduler' → expires_at > NOW() AND node_id != me → someone else is leader → wait → expires_at <= NOW() → lease expired → compete for election
    2
    ZooKeeper / etcd Ephemeral Nodes (Production-Grade)
    ZooKeeper ephemeral znodes disappear automatically when the creating session disconnects (heartbeat timeout). Nodes create sequential ephemeral znodes; lowest sequence number = leader. On leader crash, ZooKeeper deletes its znode → watch notification → next lowest becomes leader immediately. etcd equivalent: compare-and-swap on lease key with TTL. Used in Kafka (controller), HBase (master election), HDFS (NameNode HA).
    3
    Raft Consensus (Strongest — When Building Your Own System)
    Leader elected by majority vote (N/2 + 1 nodes must agree). Leader sends heartbeats every 150ms. If followers don't receive heartbeat within election timeout (300ms randomized) → start new election with incremented term. Guarantees: at-most-one leader per term (safety), progress eventually (liveness). Used in etcd, CockroachDB, TiKV. Implement only if building a DB — otherwise use etcd as a dependency.
    When to Use Each
    DB lease: simple, good for low-frequency leadership changes (cron jobs, batch jobs)
    ZooKeeper/etcd: production-grade, instant notification via watch API on leader change
    Raft: only when you're building a database or storage system that needs consensus internally
    Critical Pitfalls
    Split brain: two nodes think they're leader simultaneously → must prevent with fencing tokens
    Fencing token: monotonically increasing number attached to every write; old leader's writes rejected
    Lease too short: frequent elections; too long: slow failover on crash (30s = 30s of leaderlessness)
    ZooKeeper/etcd needs its own HA cluster (3 or 5 nodes) — don't forget to design this
    💡 Interview Pro Tip
    "For leader election, start with DB-backed lease — it's battle-tested and needs zero extra infra. Key: always check lease expiry before acting as leader (prevents split-brain from slow GC pauses causing false failover). Lease duration 30s, heartbeat every 10s gives 20s buffer. For instant failover (<1s SLA), upgrade to etcd — its watch API delivers leader-change notifications in milliseconds. Mention fencing tokens as the defense against split-brain."

    ▶ Pattern 6 — Transactional Outbox (Dual-Write Safety)
    Problem: You must write to your DB and publish an event to Kafka/SQS atomically. A crash after the DB write but before the Kafka publish = silent event loss. A crash after Kafka publish but before DB commit = duplicate event. Neither two-phase commit nor try-catch solves this reliably at scale. This is the #1 source of data inconsistency in payment systems.

    1
    Write to DB + outbox table in one transaction
    The outbox table lives in the same DB as your domain data. INSERT both in the same ACID transaction. Either both succeed or both roll back. The dual-write problem is eliminated.
    2
    Relay reads outbox → publishes to Kafka
    Two approaches:
    • Polling relay: Background thread polls outbox WHERE status='PENDING' every 100ms → publish → mark SENT. Simple but adds latency and DB load.
    • CDC (Change Data Capture): Debezium reads Postgres WAL (write-ahead log) for outbox inserts → publishes to Kafka. Near-zero overhead, sub-second latency, but operationally complex.
    3
    At-least-once delivery → consumers must be idempotent
    Relay may publish the same event twice (crash after publish, before marking SENT). Consumers must handle duplicates via idempotency key stored in Redis or DB (idempotent consumer pattern). The combination of Outbox + Idempotent Consumer = exactly-once semantics at the application level.
    Pros
    ✓ Guaranteed at-least-once delivery (no silent loss)
    ✓ No distributed transaction (2PC) needed
    ✓ Works with any message broker (Kafka, SQS, RabbitMQ)
    ✓ Audit trail: outbox is a natural event log
    ✓ Debezium CDC adds near-zero write overhead
    Cons
    ✗ Outbox table grows → needs periodic cleanup (archive/delete SENT rows)
    ✗ CDC (Debezium) is operationally complex (WAL slot management, lag monitoring)
    ✗ Polling relay adds DB query load + 100–500ms extra latency
    ✗ Consumers must implement idempotency — not free
    ✗ Schema evolution of outbox payload needs versioning (Avro / Protobuf registry)
    💡 Amazon T&PI Interview Line: "Any service that writes to a DB and publishes to a queue has a dual-write problem. I'd use the Transactional Outbox pattern: write event to an outbox table in the same DB transaction as the domain data, then use a CDC relay (Debezium on Postgres WAL) to publish to Kafka. Consumers must be idempotent. This gives at-least-once delivery with zero chance of silent event loss — which is the correct guarantee for a payment system."

    Is strong consistency required? (payments, inventory, seat booking)

    What is the primary access pattern?
    Key-value lookup (get by ID, no joins) → DynamoDB / Redis.

    Full-text search, faceted filters → Elasticsearch (mirror from primary DB, never primary store).

    Tail Latency — Why P99 Lies and P999 Matters
    The problem: A service with P50=5ms and P99=50ms looks good. But P999=500ms means 1 in every 1,000 requests takes 500ms. In a system processing 100K RPS, that's 100 slow requests per second. For a checkout flow with 5 sequential service calls, the probability all 5 are fast = 0.999⁵ ≈ 99.5% — meaning 0.5% of requests are stuck waiting for at least one slow call. This is fan-out amplification.
    Root Causes of High Tail Latency
    GC Pauses (JVM stop-the-world): Full GC can pause for 200ms–5s. P999 spikes correlate with GC events. Fix: tune G1GC/ZGC, reduce object allocation, use off-heap buffers for hot data.

    Head-of-Line (HoL) Blocking: HTTP/1.1 multiplexes one request per connection. A slow request blocks all others behind it in the queue. Fix: HTTP/2 (multiplexed streams), gRPC with concurrency.

    Lock/Mutex Contention: Shared counter, synchronized map, DB row lock. One slow lock holder delays all waiters. Fix: lock-free data structures, optimistic locking, sharded counters.

    Fan-out Amplification: Request fans out to N services/shards. P999 of result = P(at least one call slow) = 1 - (P999_per_call)^N. For N=10, P999_per_call=99.9%: final P999 = 1 - (0.999)^10 ≈ 1%. Fix: scatter-gather with timeout + best-of-N-so-far pattern.

    Cold Cache (Cache Miss Storms): After deploy or Redis restart, all requests miss cache → DB overloaded → latency spikes for 30–60s. Fix: cache warming before traffic cutover, probabilistic early expiration (PER algorithm) to avoid thundering herd.
    How to Reduce Tail Latency
    Hedged Requests: After 95th-percentile latency, send duplicate request to another replica. Use whichever returns first, cancel the other. Google paper: reduces P999 by 5× with <5% extra traffic.

    Timeout Budgets: Each hop gets a fraction of total budget. If total SLO = 200ms P99, set timeouts: API GW→Service=150ms, Service→DB=50ms. Forces failures to be fast failures (fail fast, don't queue).

    Bounded Queues: Reject requests when service is overloaded (back-pressure). A 200ms timeout in a full queue means callers wait 200ms for a 429. Bounded queue = fail fast in 1ms.

    Connection Pool Sizing: Pool too small → queue forms → latency spikes. Pool too large → DB overwhelmed. Formula: pool_size = (core_count × 2) + effective_spindle_count. PgBouncer recommended max_client_conn = DB_max_conn × 0.8.

    Read Replicas for Hot Paths: Isolate analytical/batch reads onto replica to prevent them from stealing DB connections from latency-sensitive reads.
    Fan-out P999 Math: P(request fast) = p per individual service call N = number of services in the critical path (series calls) P(ALL fast) = p^N P(at least one slow) = 1 - p^N Example: N=5 services, each p=99.9% fast P(all fast) = 0.999^5 = 99.5% P(at least one slow) = 0.5% → 1 in 200 requests hits tail latency Example: N=10 services, each p=99.9% P(all fast) = 0.999^10 = 99.0% P(at least one slow) = 1% → this is your P99, not P999! Takeaway: Keep serial service hops ≤ 3 on the critical path. Parallelize when possible. Use scatter-gather for fan-out reads.
    💡 Amazon Interview Line: "Tail latency is the hidden SLO killer. At 100K RPS, a 0.1% tail means 100 slow requests per second. For a checkout with 5 service calls, serial P999 = 1 - 0.999^5 ≈ 0.5%, so my P99 is already affected. I'd use hedged requests on the two highest-latency calls, set explicit timeout budgets per hop, and keep the critical path to ≤ 3 serial hops. I'd also alert on P999, not just P99 — Cloudwatch custom metric with 99.9th percentile."

    Say These Exactly — Phrases That Signal Senior
    These are the exact sentences. Internalize them. Vary the wording slightly but preserve the structure: threshold → consequence → my choice → trade-off stated.
    DecisionSay This
    Why SQL here "Write RPS is [X] — below the 10K threshold where a single Postgres with PgBouncer comfortably handles the load. I get ACID for free, mature tooling, and complex queries. I'd scale vertically first and add read replicas before considering a shard."
    Why NoSQL here "The access pattern is pure key-value lookup — no joins ever. Schema is fixed at design time. At [X] write RPS that exceeds what a single SQL node handles. DynamoDB gives me auto-scaling, single-digit ms reads, and I pay per request — no provisioning. The trade-off is no ad-hoc queries and no transactions across partitions — which I don't need here."
    Why cache here "Two independent signals for adding cache: hot working set exceeds DB buffer pool, and read RPS is [X]. Together they mean the DB is doing redundant disk I/O for the same rows. Redis with LRU eviction covers the top [N]% of reads. TTL set to [T] — short enough to prevent stale reads, long enough to absorb traffic spikes."
    Why Kafka not SQS "I need two things SQS doesn't give me: message replay and fan-out to multiple independent consumers. Kafka retains events for 7 days — I can reprocess for analytics without re-running the producer. Multiple consumer groups each get their own offset. The trade-off is operational complexity — I'd use MSK (managed Kafka) on AWS to reduce that."
    Why SQS not Kafka "Single consumer, no replay needed, no ordering requirement across partitions. SQS is zero operational overhead — no brokers to manage, scales to unlimited throughput, dead-letter queue built in. For async job processing with at-least-once delivery, SQS is the correct default. I'd only introduce Kafka if fan-out or replay became requirements."
    Handling duplicate events "The consumer must be idempotent. I'd store the idempotency key — typically hash of [event_id + operation_type] — in Redis with a 24-hour TTL. On each message: check Redis first. If key exists, skip processing. If not, process and set key. This gives exactly-once semantics at the application level without requiring exactly-once at the broker."
    Payments consistency "Money movement is non-negotiable for strong consistency. Eventual consistency means I might show a wrong balance — unacceptable in a payment system. I'll use Postgres with ACID transactions. For cross-service distributed transactions, I'd use the Saga pattern with compensating transactions — avoiding 2PC which is a distributed systems anti-pattern due to coordinator SPOF and blocking behavior."
    Write DB + Kafka atomically "I'd use the Transactional Outbox pattern. Write the event to an outbox table in the same DB transaction as the domain data — either both commit or both roll back. A CDC relay (Debezium on Postgres WAL) reads the outbox and publishes to Kafka. This gives at-least-once delivery with zero silent event loss. Consumers must be idempotent for the full exactly-once guarantee."
    Handling hot keys "A celebrity user with 10M followers creating a tweet creates a fan-out hotspot. Two approaches: fan-out on write (pre-populate each follower's timeline on tweet — expensive for celebrities, fast reads) vs fan-out on read (merge timeline at read time — cheaper writes, slower reads). For celebrities above [X] followers, I'd use a hybrid: fan-out on read only for accounts > 1M followers, fan-out on write for everyone else."
    Tail latency "At [X] RPS with [N] serial service calls, P999 is 1 - (0.999)^N = [Y]%. That means [Z] slow requests per second. I'd address this with: hedged requests on the two highest-latency calls — after P95 latency, send a duplicate to a second replica and use whichever returns first. I'd also set explicit per-hop timeout budgets: total SLO is [T]ms, so each hop gets T/N maximum."
